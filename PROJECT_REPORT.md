# 题目 04｜面向 RISC-V 的模拟器与调试器

## 1. 题目介绍

本项目实现了一个面向 RISC-V 的模拟器与调试器，能够加载并运行真实 ELF 程序，支持基础 RV32I 指令集、M 扩展、常见 Linux 系统调用，并提供断点、单步、寄存器查看等调试能力。

在完成基础功能的同时，项目还补充了五级流水线抽象建模、CPI 统计分析、单精度浮点（F 扩展）基础支持，以及 GDB Remote Serial Protocol（RSP）的基础接入能力，便于后续对接图形化调试器或 GDB 客户端。

## 4. 遇到的问题

1. **ELF 链接脚本问题**
   - 现象：在编译 `riscv-tests` 的 `rv32ui-v-simple` 等目标时，链接器报错 `./../env/v/link.ld:0: syntax error`，说明链接脚本并不是有效的 GNU ld 脚本，而是被误写成了类似 `../p/link.ld` 的引用文本。
   - 影响：`riscv-tests` 无法继续生成可执行文件，编译流程在链接阶段中断，前面已经生成的 `.dump` 也无法作为最终结果使用。
   - 定位方式：对比 `env/p/link.ld`、`env/pt/link.ld`、`env/pm/link.ld` 与 `env/v/link.ld` 的实际内容，确认 `v` 目录下文件内容异常。
   - 处理方式：将 `env/v/link.ld` 恢复为完整的段布局脚本，重新指定 `OUTPUT_ARCH`、`ENTRY` 和 `SECTIONS`，之后编译恢复正常并成功通过。

2. **调试器命令集不完整**
   - 现象：初始版本的调试器只能执行 `help`、`regs`、`trace`、`b`、`del`、`reset`，但题目要求的 `step`、`continue` 以及更完整的寄存器查看没有提供。
   - 影响：虽然能进入调试器，但无法对 Hello World 程序进行完整的“下断点—停住—单步—查看寄存器”流程演示。
   - 定位方式：检查 `src/debugger.cpp` 的命令分派逻辑，确认没有 `step` / `continue` 分支，同时寄存器输出格式过于简略。
   - 处理方式：补充 `step` / `s`、`continue` / `c` / `run` 命令，调用 `Simulator::step()` 和 `Simulator::run()`；同时扩展寄存器打印逻辑，将 `x0~x31` 和 PC 一起输出，满足调试要求。

3. **GDB RSP 协议接入复杂**
   - 现象：GDB 远程协议不是简单的命令字符串，而是有固定包格式、校验和、响应码和寄存器编码规则；例如寄存器需要按小端十六进制串返回，内存读写也要遵循协议包格式。
   - 影响：如果直接“按命令行方式”实现，会出现 GDB 无法识别、无法断连、无法读寄存器等问题。
   - 定位方式：梳理 GDB RSP 的最小闭环命令，优先识别 `?`、`g`、`m`、`M`、`Z0`、`z0`、`c`、`s`、`qSupported` 等常见包。
   - 处理方式：先实现 `GdbRspSession` 作为协议解析层，完成包的编解码、checksum 计算、寄存器/内存访问和断点命令，再通过 `GdbServer` 提供 socket 监听，逐步接入客户端通信。

4. **F 扩展支持范围不完整**
   - 现象：项目初始仅支持整数指令与系统调用，未提供单精度浮点寄存器和浮点运算入口。
   - 影响：无法展示浮点指令运行结果，也不利于后续对接浮点测试套件。
   - 定位方式：检查 `Simulator` 的寄存器组织与指令分派，确认 `F` 扩展尚未进入执行路径。
   - 处理方式：增加浮点寄存器槽位，并在执行器中加入 `0x53` 浮点 opcode 的处理入口，提供基础单精度浮点运算支持，为后续扩展完整测试套件预留接口。

5. **流水线统计与解释执行的统一**
   - 现象：项目主执行模型采用的是顺序解释执行，天然没有真正的 IF/ID/EX/MEM/WB 级寄存器、冒险检测和冲刷流水线；但题目又要求展示五级流水线建模和 CPI 统计。
   - 影响：如果强行改成完全流水线实现，代码规模、调试难度和出错概率都会明显上升；如果完全不做流水线抽象，又无法交付教学需要的状态图和 CPI 分析。
   - 定位方式：将“功能正确性”和“教学分析抽象”分开处理，保留解释器作为真实执行核心，再在其外层记录流水线快照和统计数据。
   - 处理方式：新增 `PipelineSnapshot` 和 `PipelineStats`，每退休一条指令就更新一次抽象流水线状态、累计周期数、指令数和 `flushes`，同时计算 CPI。这样既不破坏正确性，也能输出流水线状态变化。

6. **测试覆盖与回归验证**
   - 现象：项目同时包含模拟器核心、调试器、RSP 协议、F 扩展与 `riscv-tests` 兼容性测试，任一模块修改后都可能连带影响其他模块。
   - 影响：如果只看单个测试，很容易忽略某些边界问题，例如：断点命令改动后影响 `run_tests.sh`，或 GDB RSP 的寄存器编码与调试器寄存器打印不一致。
   - 定位方式：把测试分成“最小单元测试”和“回归测试脚本”两层；单元测试覆盖接口正确性，回归测试覆盖真实运行流程。
   - 处理方式：将 `CTest` 中的测试与 `tests/run_tests.sh` 联合起来使用；前者保证每次编译后核心模块都能跑通，后者用于批量验证 `riscv-tests`、调试器命令和异常处理路径，持续检查回归问题。

## 5. 项目具体实现

### 5.1 总体架构

项目采用“ELF 加载器 + 内存模型 + 指令解释执行 + 调试器 + 远程协议”的分层结构：

- `ELF Loader` 负责解析 ELF 文件头、程序头，并将代码段/数据段映射到模拟内存。
- `Memory` 负责字节寻址、边界检查和读写接口。
- `Simulator` 负责 CPU 状态、寄存器、PC、syscall、断点以及指令执行主循环。
- `Debugger` 负责命令行交互，提供断点、单步、继续运行、寄存器查看等功能。
- `GDB RSP` 模块提供基础远程调试协议封装，方便后续对接 GDB。
- `GdbServer` 提供 socket 监听与客户端通信骨架，支持外部调试器接入。

### 5.2 ELF 加载与内存映射

程序启动时，`Simulator::load_program()` 调用 ELF 加载器读取目标文件，并将可执行段映射到内存中。项目支持对程序入口点、`tohost` / `fromhost` 符号地址的识别，从而能够正确启动测试程序并与测试框架交互。

内存方面，当前实现使用连续字节数组作为主存，并通过 `MemoryRegion` 管理可访问区域。这样既能模拟代码段、数据段、栈段等区域，也能在访问非法地址时立即返回错误，便于调试越界问题。

### 5.3 指令译码与执行

指令译码在 `riscv_isa.hpp` 中通过位操作完成：

- `opcode = inst[6:0]`
- `rd = inst[11:7]`
- `funct3 = inst[14:12]`
- `rs1 = inst[19:15]`
- `rs2 = inst[24:20]`
- `funct7 = inst[31:25]`

`decode()` 返回 `DecodedInst` 结构体，供执行器使用。执行阶段在 `Simulator::execute_one()` 中完成，当前支持：

- RV32I 基础整数运算和控制转移
- Load / Store 指令
- `ECALL` / `EBREAK`
- M 扩展中的乘除法相关操作
- F 扩展中的基础单精度浮点运算入口

执行完成后会自动保持 `x0 = 0`，并更新 PC 和寄存器状态。

### 5.4 系统调用模拟

项目模拟了常见 Linux syscall：

- `write`
- `read`
- `exit`
- `brk`

其中：

- `write` 将内存中的字符串输出到标准输出
- `read` 从标准输入读取数据并写回模拟内存
- `exit` 设置退出码并停止模拟器
- `brk` 用于管理堆区边界，配合内存映射模拟动态内存增长

### 5.5 调试器实现

调试器通过 `Debugger::handle_command()` 解析命令，支持：

- `b <addr>`：设置断点
- `del <n>`：删除断点
- `step` / `s`：单步执行
- `continue` / `c` / `run`：继续运行
- `regs` / `r`：打印 PC 和全部通用寄存器
- `pipeline` / `pipe` / `cpi`：查看流水线快照与 CPI 统计
- `trace on/off`：开关指令跟踪
- `reset`：重置模拟器

在 Hello World 程序中，可以先用 `b` 在入口或 `main` 附近下断点，再用 `continue` 运行到断点，随后使用 `step` 逐条执行并用 `regs` 查看寄存器变化。

### 5.6 五级流水线抽象与 CPI 统计

虽然当前核心执行模型仍以顺序解释执行保证正确性，但项目增加了五级流水线抽象：

- IF
- ID
- EX
- MEM
- WB

每次指令退休后更新一次流水线快照，并维护 `PipelineStats`：

- `cycles`
- `instructions`
- `stalls`
- `flushes`
- `cpi`

这样既保留了教学场景中对流水线各级寄存器的描述，也能统计控制流变化带来的冲刷开销，为后续进一步实现真实 hazard 检测逻辑预留接口。

### 5.7 F 扩展基础支持

项目增加了单精度浮点支持的最小实现：

- 新增 32 个浮点寄存器槽位
- 新增浮点运算入口 `0x53`
- 支持基础单精度运算：加、减、乘、除、平方根、最小值和最大值

该部分主要用于说明模拟器已具备扩展至浮点测试套件的能力，并可作为后续补齐完整 F 扩展指令集的基础。

### 5.8 GDB RSP 基础支持

`GdbRspSession` 实现了基础远程调试协议处理，包括：

- 读取寄存器 `g`
- 读取/写入内存 `m` / `M`
- 设置/删除断点 `Z0` / `z0`
- 继续运行 `c`
- 单步 `s`
- 查询支持能力 `qSupported`

`GdbServer` 则负责 socket 监听与会话转发，使模拟器可以作为远程调试目标被 GDB 客户端连接。该实现为后续接入标准远程调试环境打下了基础，也方便把本项目从命令行调试扩展到标准远程调试环境。

### 6.5 如何理解这个项目的具体技术点

- **ELF 解析**：负责识别程序入口、段地址和符号信息，使真实二进制可以直接加载。
- **位域解码**：通过按位截取 `opcode`、`rd`、`funct3`、`rs1`、`rs2`、`funct7` 完成指令识别。
- **内存模型**：模拟代码段、数据段和栈堆区域，保证访存越界可被检测。
- **系统调用模拟**：把程序里的 `write/read/exit/brk` 映射到宿主机行为。
- **调试器**：提供断点、单步和寄存器检查，帮助观察程序状态。
- **五级流水线抽象**：提供 IF/ID/EX/MEM/WB 的状态快照，便于分析控制流变化。
- **CPI 分析**：统计周期、指令数和 flush 事件，帮助评估执行效率。
- **F 扩展**：增加基础单精度浮点运算入口，为进一步扩展浮点指令集预留空间。
- **GDB RSP**：用标准远程协议格式封装调试能力，方便接入 GDB 客户端。

### 6.6 团队如何正确使用 AI

本项目中，AI 主要作为“辅助设计、辅助查错、辅助整理”的工具，而不是直接替代人工决策。正确使用方式包括：

- 先由成员阅读题目要求和代码结构，再让 AI 帮助梳理实现方案
- 让 AI 生成测试思路、边界用例和文档草稿
- 对 AI 给出的代码和结论进行人工复核，避免直接照搬
- 对关键逻辑进行二次验证，例如编译、单测、回归测试
- 当 AI 输出与规范不一致时，优先以项目代码、官方文档和测试结果为准

本项目在实际开发中也会“纠正 AI”生成内容中的问题，例如：

- 当协议或链接脚本内容不正确时，依据编译报错回改
- 当调试器功能缺失时，按验收标准补齐而不是接受不完整结果
- 当 AI 方案与真实代码冲突时，以实际可运行、可测试的实现为准
